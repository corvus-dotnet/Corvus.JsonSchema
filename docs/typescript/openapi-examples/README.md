# TypeScript Patterns - OpenAPI Client Generation

This recipe demonstrates how to generate and use a strongly-typed **TypeScript** HTTP client from an
[OpenAPI](https://spec.openapis.org/oas/latest.html) specification using
`corvusjson openapi-client --engine TypeScript`. The example uses the Petstore-style API in three flavours —
`petstore-3.0/`, `petstore-3.1/`, and `petstore-3.2/` — so the same client surface is exercised across every
supported OpenAPI version. For the full reference, see the [OpenAPI client generation](../openapi.md) guide.

> **Version compatibility:** the generator auto-detects the spec version from the `openapi` field. Request
> composition and response decomposition are version-invariant, so the three clients produce identical wire.

## What the Generator Does For You

When you run `corvusjson openapi-client --engine TypeScript`, the generator reads your OpenAPI spec and
produces:

| Generated artifact | What it handles |
|---|---|
| **Client class** (`ApiStatusClient`) | One async method per operation: builds the request, sends it via the injected transport, returns the typed response. |
| **Request modules** (`getPetRequest`, …) | Serialize path / query / header / cookie parameters into the correct wire format (percent-encoding, form / matrix / label / deepObject / space- and pipe-delimited styles) and carry the request body. |
| **Response classes** (`GetPetResponse`, …) | Decode the response bytes into typed models; expose `tryGet{Status}()`, an exhaustive `match({…})`, typed response-header getters, and `validate()`. |
| **Models** (`Pet`, `PetUpdate`, `Error`) | JSON Schema types (validators + parsers + builders) generated by the TypeScript model engine, imported from `./models`. |
| **Interface** (`IApiStatusClient`) | Contract for injecting a mock transport in tests. |

The generated code handles parameter serialization (byte-for-byte identical to the C# client), request
validation *before* the call leaves your process, response parsing into typed models, typed response headers
(returned as their schema's model type), streaming bodies as `AsyncIterable`, and response links.

## The Spec

Each `petstore-*/openapi.json` describes a small pet API — path/query/header/cookie parameters across every
style, JSON / form-urlencoded / multipart / octet-stream / text request bodies, typed response headers,
streaming responses, and response links — enough to exercise the full generated client surface.

## Generating the Client

```bash
corvusjson openapi-client petstore-3.0/openapi.json \
    --engine TypeScript \
    --outputPath petstore-3.0/client \
    --tsClientRuntimeModule "@endjin/corvus-json-client-runtime"
```

`regenerate.sh` runs this for all three specs and strict-type-checks the output with `tsc`.

## Using the Client

```typescript
import { FetchApiTransport, ValidationMode } from "@endjin/corvus-json-client-runtime";
import { ApiStatusClient } from "./petstore-3.0/client/ApiStatusClient.js";

await using transport = new FetchApiTransport({ baseUrl: "https://api.example.com/v1" });
const client = new ApiStatusClient(transport);

// Path + query + header params compose in one call; a compound array is percent-encoded per its style.
const response = await client.updatePet(
  { petId: "p 1", tags: ["a b", "a/b"], verbose: true, xRequestId: "req-1" },
  { name: "Rex", tag: "dog" },
);

// Exhaustive status handling; each case gets its decoded body.
response.match({
  ok: (pet) => console.log(`updated ${pet.name}`),
  default: (error) => console.error(`error ${error.code}: ${error.message}`),
});

// Optional: assert the body conforms before trusting it.
response.validate(ValidationMode.Basic);

// Follow a declared response link (petId resolved from the response body):
const pet = await response.links.getPet();
```

## Running the Harness

Two scripts drive the recipe end to end:

- **`regenerate.sh`** — builds the CLI, regenerates each `petstore-*/client`, and strict-type-checks with
  `tsc` (the load-bearing gate: generated client that does not type-check fails the build).
- **`conformance.sh`** — regenerates each client with the published package-name runtime import, compiles,
  and runs the `node --test` conformance + C#↔TypeScript **parity** suites (`conformance.test.mjs`,
  `parity.test.mjs`) — asserting the TypeScript client composes byte-identical request wire and decodes
  responses identically to the C# client, over the shared `parity/` fixtures.

```bash
bash regenerate.sh   # generate + strict tsc
bash conformance.sh  # generate + compile + node:test conformance & parity
```

## Regeneration

The generated `petstore-*/client` trees are checked in and regenerated by `regenerate.sh` after any change
to the generator or the specs. The runtime is imported by the **package name**
(`@endjin/corvus-json-client-runtime`) in shipped clients; the in-tree type-check uses a relative path to the
built package only so the recipes type-check without a published dependency.
