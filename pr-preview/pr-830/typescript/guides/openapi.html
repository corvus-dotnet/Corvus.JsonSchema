OpenAPI client generation

Try the OpenAPI playground — generate a strongly-typed TypeScript client from a spec, inspect the request/response modules, and run it against a mock transport in your browser.

This is the reference for generating a TypeScript HTTP client from an OpenAPI specification: the corvusjson openapi-client --engine TypeScript command, the shape of the output, the byte-native transport runtime, and how you consume the generated client. For a first walkthrough of the TypeScript engine, start with Getting Started; for the model-generation reference, see Code generation.

Overview

The generator reads an OpenAPI 3.0, 3.1, or 3.2 specification and writes a strongly-typed HTTP client in plain TypeScript. The same engine backs the C# generator — the --engine TypeScript option selects the TypeScript emitter — so the two clients compose byte-identical wire requests and decode responses identically (this is enforced by a C#↔TypeScript parity harness).

The generated code handles all the HTTP plumbing — parameter serialization, JSON Schema validation, request/response parsing, streaming, and error handling — so you focus purely on business logic. It targets fetch (browsers, edge runtimes, Node) with an optional Node/undici transport, depends on no framework, and never touches .NET at run time: the output is TypeScript plus a small runtime library.

Server generation is C#-only. The TypeScript engine emits clients; use the C# openapi-server command for ASP.NET Core server stubs (see OpenAPI Code Generation).

Installation

The generator is a .NET global tool and requires the .NET SDK:

dotnet tool install --global Corvus.Json.Cli

Your generated client and your project consume two npm packages — the model runtime and the client (transport) runtime:

npm install @endjin/corvus-json-runtime @endjin/corvus-json-client-runtime

@endjin/corvus-json-runtime provides the zero-dependency JSON Schema model runtime (validation, builders); @endjin/corvus-json-client-runtime provides the byte-native transport, parameter serializers, middleware, and authentication. Both are ESM, side-effect-free, and tree-shakeable.

Quick Start

Generate a typed client from any OpenAPI spec:

corvusjson openapi-client petstore.json \
    --engine TypeScript \
    --outputPath ./src/api

Use it with a few lines:

import { FetchApiTransport, ValidationMode } from "@endjin/corvus-json-client-runtime";
import { ApiPetsClient } from "./api/ApiPetsClient.js";

await using transport = new FetchApiTransport({ baseUrl: "https://petstore.example.com/v1" });
const client = new ApiPetsClient(transport);

// List pets — the generated code validates limit <= 100 before the request leaves your process.
const response = await client.listPets({ limit: 10 });

response.match({
  ok: (pets) => {
    for (const pet of pets) {
      console.log(`[${pet.id}] ${pet.name}`);
    }
  },
  default: (error) => {
    console.error(`Error ${error.code}: ${error.message}`);
  },
});

The client owns the transport's lifetime via Symbol.asyncDispose; await using releases the underlying sockets when the scope exits. The response body is buffered and decoded lazily through the model companions.

What the Generator Produces

For each operation the generator reads your spec and emits four kinds of module:

Generated artifact What it handles
Client class (ApiPetsClient) One async method per operation. Builds the request, sends it via the injected transport, and returns the typed response.
Request modules (listPetsRequest, …) Serialize path / query / header / cookie parameters into the correct wire format (percent-encoding per RFC 3986, form / matrix / label / deepObject / space- and pipe-delimited styles) and carry the request body.
Response classes (ListPetsResponse, …) Decode the response bytes into typed models. Expose tryGet{Status}() and an exhaustive match({…}) for status-code handling, typed response-header getters, and validate().
Models (Pet, NewPet, Error) JSON Schema types generated by the TypeScript model engine — validators, parsers, and builders, imported from ./models.

Every operation also gets an interface (IApiPetsClient) so you can inject a mock in tests, and the client exposes a static serverUri(...) factory built from the spec's servers block (with server-variable substitution).

Generated Code Architecture

Client flow

The generated client is a thin orchestration layer over an injectable ApiTransport:

client.listPets(params)
  → listPetsRequest(params)          // an ApiRequest: writeResolvedPath / writeQueryString / writeHeaders
  → transport.send(request, factory) // composes the wire request, runs middleware + auth, sends
  → listPetsResponseFactory.create(context)
  → ListPetsResponse                 // tryGet* / match / validate / header getters

The transport consumes the request's write* methods synchronously into a reusable byte buffer and snapshots the wire request before the first await, so request composition allocates minimally and never races an in-flight mutation. The request body is a discriminated union — none, bytes, stream, or writer (multipart) — and is materialized only at dispatch.

Parameter Styles

Every OpenAPI parameter style is generated as byte-native serialization, matching the C# client byte-for-byte:

Location Styles
Path simple, label, matrix (× explode)
Query form (× explode), spaceDelimited, pipeDelimited, deepObject
Header simple
Cookie form

Reserved characters in parameter values are percent-encoded (/ → %2F, space → %20); array and object values are serialized per their style — for example a form non-explode array joins with , (tags=a%20b,a%2Fb), a deepObject filter emits filter[min]=1&filter[max]=2, and a matrix object path segment emits ;scope=kind,k1,region,r1. Booleans serialize as lowercase true/false.

// scope (matrix object) + tags (spaceDelimited) + filter (deepObject) all compose in one call:
await client.search({
  scope: { kind: "cat", region: "eu" },
  tags: ["indoor", "long hair"],
  filter: { minAge: "1", maxAge: "9" },
});

Response Handling

Exhaustive matching

match({…}) dispatches on the response status and hands each case its decoded body. Cases are named by the response key — ok for a 2xx success, the status accessor name for others, and default for the default response — with an otherwise fallback:

const label = response.match({
  ok: (pet) => `found ${pet.name}`,
  default: (error) => `error ${error.code}`,
  otherwise: (status) => `unexpected ${status}`,
});

For simple checks, tryGet{Status}() returns the decoded body or undefined:

const pet = response.tryGetOk();
if (pet !== undefined) {
  console.log(pet.name);
}

Response headers

Declared response headers are exposed as typed getters that return the header schema's model type and validate the value against it. A plain scalar returns number / string; a formatted scalar returns the branded type (a date-time header returns Brand<string, "date-time">, on which the model's toTemporal accessor yields a Temporal.Instant); an object header returns the typed interface:

const remaining = response.xRateLimitHeader;      // number | undefined, validated
const expiresAt = response.xExpiresAtHeader;      // a branded date-time; XExpiresAt.toTemporal(expiresAt)

A getter throws if the header value fails its schema; an absent header is undefined.

Request Bodies

The generator handles every request body category, choosing the right runtime serializer:

// JSON — built and validated through the model companion:
await client.createPet({ name: "Luna", tag: "cat" });

// application/x-www-form-urlencoded:
await client.submitForm({ name: "gadget", count: 3, tags: ["a", "b"] });

// text/plain:
await client.postNote("hello note");

// application/octet-stream (raw bytes or a stream):
await client.upload(new TextEncoder().encode("binary-data"));

// multipart/form-data — typed fields + a binary part:
await client.uploadAvatar(
  { id: "a1", tags: ["t1", "t2"] },
  { content: fileBytes, filename: "avatar.png", contentType: "image/png" },
);

JSON bodies are built and validated by the model type before sending; form, multipart, and octet-stream bodies stream their bytes into the transport's buffer with no intermediate string.

Streaming Responses

Streaming operations (OpenAPI 3.2 itemSchema, text/event-stream and application/x-ndjson) expose the body as an AsyncIterable, so you consume items as they arrive rather than buffering the whole response:

// NDJSON — one typed item per line:
for await (const activity of response.enumerateOkItems()) {
  console.log(activity.kind);
}

// Server-Sent Events — typed data plus event metadata:
for await (const event of response.enumerateOkSseItems()) {
  console.log(event.event, event.data);
}

Streaming statuses omit tryGet/match for that case — the body is consumed through the iterator.

Validation

Response bodies validate against their JSON Schema on demand. validate(mode) throws on failure; None (the default) skips validation, Basic runs a boolean evaluate, Detailed collects per-keyword failures:

import { ValidationMode } from "@endjin/corvus-json-client-runtime";

response.validate(ValidationMode.Basic); // throws if the active body fails its schema

Request parameters and JSON bodies are always validated against their schema before the HTTP call leaves your process, so an invalid request never reaches the network.

Authentication

Pass an AuthenticationProvider to the transport; it runs once per logical send, after the request is built and before middleware:

import { FetchApiTransport, bearerToken, apiKey, basicAuth } from "@endjin/corvus-json-client-runtime";

// Bearer token (static or refreshed per request via a factory):
new FetchApiTransport({ baseUrl, authenticationProvider: bearerToken("eyJ…") });
new FetchApiTransport({ baseUrl, authenticationProvider: bearerToken(async () => getFreshToken()) });

// API key in a header, query parameter, or cookie:
new FetchApiTransport({ baseUrl, authenticationProvider: apiKey("secret", "X-Api-Key", "header") });

// HTTP Basic:
new FetchApiTransport({ baseUrl, authenticationProvider: basicAuth("user", "pass") });

Middleware and Resilience

The transport runs a composable chain of wire-level Handlers over WireRequest → Promise<WireResponse>. The built-ins cover retry (with Retry-After + backoff + jitter), redirect following (auth stripped on cross-origin), and per-attempt timeouts:

import { FetchApiTransport, retryHandler, redirectHandler, timeoutHandler } from "@endjin/corvus-json-client-runtime";

new FetchApiTransport({
  baseUrl,
  handlers: [
    redirectHandler({ maxRedirects: 5 }),
    retryHandler({ maxAttempts: 3 }),
    timeoutHandler({ perAttemptMs: 10_000 }),
  ],
});

defaultHandlers({...}) assembles the recommended order (redirect → retry → timeout → send) in one call.

Transports

Two transports ship with the runtime, sharing identical middleware and auth — only the dispatch differs:

  • FetchApiTransport (the default barrel, @endjin/corvus-json-client-runtime) — for browsers, edge runtimes, and Node; owns redirect handling (redirect: "manual").
  • NodeApiTransport (@endjin/corvus-json-client-runtime/node) — an undici-based transport for Node, with a socket-dispose owner and an http fallback.
import { NodeApiTransport } from "@endjin/corvus-json-client-runtime/node";
await using transport = new NodeApiTransport({ baseUrl: "https://petstore.example.com/v1" });

Links

Response links (OpenAPI runtime expressions) are exposed on response.links, which resolves the target operation's parameters from the response and the originating request and invokes it via the same transport. The generated followers resolve $response.body#/…, $response.header.…, $request.path/query/header.…, and $request.body#/… (matching the C# client; $url and $method are unsupported in both):

const created = await client.updatePet({ petId: "p1", xRequestId: "r1" }, { name: "Rex" });
// Follow the declared link — petId is resolved from the response body's id:
const pet = await created.links.getPet();

CLI Reference

corvusjson openapi-client <spec> --engine TypeScript [options]
Option Description
--engine TypeScript Select the TypeScript emitter (default is V5, idiomatic C#).
--outputPath <dir> Where to write the generated client. Models are written under <dir>/models.
--clientName <name> Override the generated client class name.
--tsClientRuntimeModule <spec> The transport-runtime import specifier (default @endjin/corvus-json-client-runtime).
--tsRuntimeModule <spec> The model-runtime import specifier.
--tsModulePerType Emit one module per type (default is a single models/generated.ts barrel).
--spec-url <url> Fetch the spec from a URL (recorded for re-fetch).
--yaml Parse the spec as YAML.
--force Regenerate even when the lock file reports no changes.

Use corvusjson openapi-show <spec> to inspect the operations, and the --group-by / path-filter options to generate a subset of a large spec.

Version Support

The generator auto-detects the spec version from the openapi field and supports OpenAPI 3.0, 3.1, and 3.2 from one emitter. Request composition and response decomposition are version-invariant — a client generated from any of the three versions produces the same wire for the same inputs.

Best Practices

  • Inject the transport. Take ApiTransport (or the generated IApiPetsClient interface) in your code so tests can substitute a mock; the conformance suite drives the generated clients through an in-memory transport this way.
  • Dispose the client. Use await using (or call Symbol.asyncDispose) so the transport releases sockets.
  • Validate at the boundary. Requests validate before sending; call response.validate(ValidationMode.Basic) when you need to assert a response conforms before trusting it.
  • Keep the runtime import as the package name in shipped code (@endjin/corvus-json-client-runtime) so it resolves from node_modules; the in-repo examples use a relative path only for in-tree type-checking.
